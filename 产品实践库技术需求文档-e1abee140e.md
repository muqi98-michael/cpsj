## 1. 项目概述

### 1.1 项目背景
当前企业面临知识分散、检索困难、一线赋能不足的核心问题。产品解决方案（PPT/Word）散落在各处，缺乏结构化管理；一线人员（销售、售前、交付）难以快速找到匹配客户场景的解决方案；知识积累与实际应用脱节，总部与一线存在信息断层。

本项目旨在构建一个"产品实践库"网站，通过结构化和打标签的方式存放产品解决方案文档，让一线人员通过搜索、知识地图、AI问答等方式方便地获取知识库内容，实现总部知识向一线的有效传递。

### 1.2 项目目标
- 构建场景-解决方案-案例三位一体的知识闭环体系
- 实现80%以上的产品解决方案完成结构化入库
- 搜索命中率>85%，AI问答满意度>80%
- 一线人员月活跃率>70%
- 同一解决方案被引用次数>10次/月

### 1.3 用户群体

| 用户角色 | 描述 | 主要需求 |
|---------|------|---------|
| 总部方案顾问 | 知识生产者，懂产品、懂业务 | 便捷的结构化录入工具，系统化组织知识 |
| 一线销售 | 需要快速响应客户，场景化查询 | 通过关键词或场景快速定位解决方案，获取话术支持 |
| 一线售前 | 需要深度产品知识，方案定制化 | 基于客户场景检索相关方案和案例，获取产品特性细节 |
| 一线交付/实施 | 需要落地指导，问题解决 | 获取最佳实践、问题处理方法，快速上手项目 |
| 管理员 | 系统运维，数据治理 | 基础数据维护、权限管理、数据质量控制 |

### 1.4 使用场景

**场景1：销售快速匹配客户场景**
- 使用者：一线销售
- 触发条件：与客户沟通时，客户描述某个业务痛点
- 操作流程：
  1. 在首页搜索框输入关键词（如"库存管理"）
  2. 系统返回匹配的业务场景、解决方案和相关案例
  3. 点击查看详情，下载PPT或直接AI问答获取关键话术
- 预期效果：30秒内找到匹配方案，提升客户沟通效率

**场景2：方案顾问知识录入与关联**
- 使用者：总部方案顾问
- 触发条件：有新的产品解决方案需要入库
- 操作流程：
  1. 进入后台，创建/选择业务场景
  2. 上传PPT/Word文档，系统自动解析元数据
  3. 关联已有解决方案和客户案例
  4. 设置标签、分类、权限
- 预期效果：知识结构化入库，形成场景→方案→案例的闭环

**场景3：一线人员AI问答获取知识**
- 使用者：一线售前/交付
- 触发条件：遇到具体问题需要快速解答
- 操作流程：
  1. 在首页AI问答框输入问题（如"制造业的生产计划场景有哪些核心特性？"）
  2. 系统基于向量检索返回相关内容
  3. AI生成结构化答案，附带来源链接
- 预期效果：无需人工翻阅文档，快速获得精准答案

### 1.5 成功标准
- [ ] 知识入库覆盖率：80%以上的产品解决方案完成结构化入库
- [ ] 检索准确率：关键词检索命中率>85%，AI问答满意度>80%
- [ ] 使用活跃度：一线人员月活跃率>70%
- [ ] 知识复用率：同一解决方案被引用次数>10次/月

## 2. 功能需求

### 2.1 功能模块架构

系统包含以下核心模块：
- **首页浏览模块**：知识地图、最新/最热内容、搜索入口、AI问答入口
- **基础数据管理模块**：产品、行业、角色等维度数据管理
- **场景库管理模块**：业务场景的定义、分类、标签、权限管理
- **解决方案库管理模块**：解决方案文档上传、元数据管理、关联场景
- **客户案例库管理模块**：典型案例录入、关联场景和解决方案
- **知识地图配置模块**：知识地图层级结构、卡片样式配置
- **搜索功能模块**：全文检索、标签筛选、多维度组合查询
- **AI问答模块**：基于RAG的智能问答，支持5轮上下文
- **API开放模块**：RESTful API，供内部官网等外部系统调用
- **MCP协议模块**：支持公司AI主程序调用本产品库Agent

### 2.2 核心功能详细设计

#### 功能模块1：首页浏览

**功能描述**
作为知识库的统一入口，为总部和一线用户提供直观的知识浏览、检索和AI问答能力。核心是通过知识地图的可视化呈现，让用户能够快速定位到感兴趣的业务场景和解决方案。

**用户故事**
```
作为一线销售，
我希望通过知识地图快速浏览所有业务场景，
以便在客户沟通前快速了解产品覆盖的业务范围。

作为一线售前，
我希望看到最新和最热门的解决方案，
以便及时掌握最新的产品能力。
```

**输入定义**
- 数据来源：从业务数据库读取场景库、解决方案库、案例库的元数据
- 热度算法：基于浏览量、下载量、AI问答调用次数计算热度分
- 更新频率：最新内容实时更新，热度榜单每小时更新

**输出定义**
- 结果形式：首页页面，包含三个核心区域
  - 顶部搜索栏 + AI问答入口
  - 知识地图区域（卡片式分类展示）
  - 最新/最热内容列表
- 时效要求：首屏加载<2秒，搜索响应<500ms
- 异常处理：数据加载失败时显示降级UI（静态提示）

**交互流程**
1. 用户访问首页URL
2. 系统根据用户角色加载个性化内容（一线用户优先展示相关行业场景）
3. 知识地图采用卡片式布局，支持展开/折叠二级分类
4. 点击场景卡片进入场景详情页
5. 点击解决方案卡片跳转至解决方案详情页
6. 搜索框支持实时联想（输入2个字符后触发）

**验收标准**
- [ ] 知识地图支持3级分类展开
- [ ] 搜索框输入2个字符后触发联想
- [ ] 最新/最热列表支持分页，每页10条
- [ ] AI问答入口在首屏可见区域
- [ ] 移动端适配，知识地图支持横向滚动

#### 功能模块2：基础数据管理

**功能描述**
由系统管理员统一维护产品、行业、角色等维度的基础数据，这些数据是整个知识库的分类基础和标签来源。

**用户故事**
```
作为系统管理员，
我希望能够统一维护产品、行业、角色等基础数据，
以便保证知识分类的一致性和规范性。

作为方案顾问，
我希望在录入解决方案时能够直接选择已有的产品/行业标签，
以便提高录入效率并避免手动输入错误。
```

**输入定义**
- 数据来源：管理员手动录入或批量导入Excel
- 格式要求：
  - 产品数据：产品名称、产品编码、所属产品线、状态（启用/停用）
  - 行业数据：行业名称、行业编码、父级行业（支持多级）
  - 角色数据：角色名称、角色编码、所属部门
- 校验规则：编码唯一性校验、名称非空校验

**输出定义**
- 结果形式：后台管理页面，包含数据列表和表单
- 存储位置：基础数据表（product_master、industry_master、role_master）
- 关联影响：删除操作需检查是否被业务数据引用

**交互流程**
1. 管理员进入"基础数据管理"菜单
2. 选择要管理的数据类型（产品/行业/角色）
3. 查看现有数据列表，支持搜索和排序
4. 点击"新增"打开表单，填写必填字段
5. 点击"编辑"修改已有数据
6. 点击"删除"前检查引用关系，如有引用则禁止删除

**验收标准**
- [ ] 产品支持产品线多级分类（最多3级）
- [ ] 行业支持多级分类（省-市-行业细分）
- [ ] 角色数据支持与部门系统的集成（预留接口）
- [ ] 所有操作记录审计日志
- [ ] 支持批量导入导出Excel

#### 功能模块3：场景库管理

**功能描述**
产品场景库是三大业务库的核心，用于定义客户的业务场景。每个场景包含编号、简介、痛点、角色、目标、关联产品、关联解决方案等结构化信息。

**用户故事**
```
作为方案顾问，
我希望能够创建和维护业务场景的定义，
以便为解决方案和案例提供结构化的场景基础。

作为一线销售，
我希望能够按照行业或角色快速筛选场景，
以便快速找到匹配客户背景的解决方案。
```

**输入定义**
- 数据来源：方案顾问手动录入
- 格式要求：
  - 场景编号：自动生成（SCN-YYYYMMDD-序号）
  - 场景简介：500字以内
  - 场景痛点：多行文本，支持分点
  - 涉及角色：多选（从基础数据中选择）
  - 场景目标：多行文本
  - 关联产品：多选（从产品库中选择）
  - 关联解决方案：多选（从解决方案库中选择）
  - 标签：支持自定义标签，也支持从基础数据中选择
  - 权限：设置查看权限（公开/部门可见/指定角色可见）
- 校验规则：场景编号唯一、简介和痛点必填

**输出定义**
- 结果形式：场景详情页、场景列表页
- 存储位置：场景库表（business_scenario）
- 关联影响：场景删除需同步解除与解决方案、案例的关联

**交互流程**
1. 方案顾问进入"场景库管理"菜单
2. 查看场景列表，支持按行业、产品、标签筛选
3. 点击"新增场景"打开编辑表单
4. 填写场景基本信息（编号自动生成）
5. 选择涉及角色、关联产品
6. 关联已有解决方案（可搜索选择）
7. 设置标签和权限
8. 保存后场景立即可见（如权限允许）
9. 场景支持版本管理，修改时保留历史版本

**验收标准**
- [ ] 场景编号自动生成，格式为SCN-YYYYMMDD-001
- [ ] 关联解决方案支持模糊搜索和多选
- [ ] 场景详情页展示完整的元数据
- [ ] 支持场景的启用/停用状态切换
- [ ] 场景删除需要二次确认
- [ ] 场景列表支持导出Excel

#### 功能模块4：解决方案库管理

**功能描述**
解决方案库存储SaaS厂商对应业务场景提供的产品解决方案，以PPT、Word等文档形式存在，同时具备结构化的元数据（应用价值、主要特性、关联场景、关联流程等）。

**用户故事**
```
作为方案顾问，
我希望能够上传解决方案文档并填写结构化的元数据，
以便一线人员能够通过文档内容和元数据双维度检索。

作为一线售前，
我希望能够在线预览PPT内容，而不需要下载，
以便快速筛选解决方案。
```

**输入定义**
- 数据来源：方案顾问上传文档 + 手动填写元数据
- 格式要求：
  - 解决方案名称：必填
  - 文档上传：支持PPT、Word、PDF，单文件最大100MB
  - 应用价值：多行文本，分点描述
  - 主要产品特性：多行文本，结构化分点
  - 关联业务场景：多选（从场景库中选择）
  - 关联流程：可选，关联流程图或流程说明
  - 标签：多选
  - 版本号：手动填写（v1.0, v1.1等）
  - 权限：设置查看和下载权限
- 校验规则：文档必须上传、名称必填

**输出定义**
- 结果形式：解决方案列表页、详情页、文档预览页
- 存储位置：
  - 元数据：解决方案库表（solution_library）
  - 文档：对象存储（OSS）
  - 向量数据：上传后自动向量化存入向量数据库
- 关联影响：解决方案删除需同步解除与场景、案例的关联

**交互流程**
1. 方案顾问进入"解决方案库管理"菜单
2. 点击"新增解决方案"
3. 填写解决方案名称、版本号
4. 上传文档（支持拖拽上传）
5. 系统自动解析文档元数据（标题、作者等）
6. 填写应用价值、主要产品特性
7. 选择关联的业务场景
8. 设置标签和权限
9. 保存后系统自动触发文档向量化任务
10. 向量化完成后，解决方案可被AI检索

**验收标准**
- [ ] 文档上传支持断点续传
- [ ] 文档上传后自动触发向量化（异步任务）
- [ ] 文档支持在线预览（采用第三方服务或基础HTML渲染）
- [ ] 文档支持版本管理，保留历史版本
- [ ] 解决方案列表支持按时间、热度、相关性排序
- [ ] 下载操作记录审计日志

#### 功能模块5：客户案例库管理

**功能描述**
客户案例库存储场景的典型案例，包含客户基本信息、场景描述、解决方案实施过程、实施效果等，用于一线人员的案例参考和说服客户。

**用户故事**
```
作为方案顾问，
我希望能够录入客户案例并关联对应的场景和解决方案，
以便一线人员能够用真实的成功案例说服客户。

作为一线销售，
我希望能够按照行业和场景快速查找案例，
以便在客户沟通时提供同行业成功案例。
```

**输入定义**
- 数据来源：方案顾问手动录入
- 格式要求：
  - 案例标题：必填
  - 客户名称：必填（可匿名化处理）
  - 客户行业：从基础数据中选择
  - 案例描述：详细描述客户背景和问题
  - 实施方案：描述解决方案的实施过程
  - 实施效果：量化效果（如效率提升30%、成本降低20%）
  - 关联业务场景：多选
  - 关联解决方案：多选
  - 案例文档：可选，上传详细案例文档
  - 权限：设置查看权限（公开/部门可见/私有）
- 校验规则：标题、客户名称、行业必填

**输出定义**
- 结果形式：案例列表页、案例详情页
- 存储位置：案例库表（customer_case）
- 关联影响：案例删除需同步解除与场景、解决方案的关联

**交互流程**
1. 方案顾问进入"客户案例库管理"菜单
2. 点击"新增案例"
3. 填写案例基本信息（标题、客户名称、行业）
4. 填写案例描述、实施方案、实施效果
5. 选择关联的业务场景和解决方案
6. 上传案例文档（可选）
7. 设置权限
8. 保存后案例立即可见

**验收标准**
- [ ] 案例支持按行业、场景、解决方案筛选
- [ ] 案例详情页展示完整信息
- [ ] 案例支持匿名化客户名称
- [ ] 案例列表支持导出Excel
- [ ] 实施效果支持数据展示（如进度条展示提升比例）

#### 功能模块6：知识地图配置

**功能描述**
管理员可以对产品场景库进行知识地图配置，以分类卡片地图的形式呈现，方便用户浏览。支持自定义地图的层级结构、卡片样式、展示维度等。

**用户故事**
```
作为管理员，
我希望能够配置知识地图的层级结构和展示方式，
以便不同行业/部门的用户能够看到定制化的知识地图。

作为一线用户，
我希望知识地图的卡片能够直观展示场景的关键信息，
以便快速了解场景内容。
```

**输入定义**
- 数据来源：管理员手动配置
- 格式要求：
  - 地图名称：如"制造业知识地图"、"零售业知识地图"
  - 地图类型：按行业/按产品/按角色
  - 层级结构：最多3级（根节点-分类-场景）
  - 卡片展示字段：选择场景在卡片上展示的字段（标题、简介、热度等）
  - 排序规则：按热度/按时间/按名称
  - 适用角色：选择可见该地图的用户角色
- 校验规则：地图名称唯一、层级结构必须有效

**输出定义**
- 结果形式：知识地图配置页、前端地图渲染
- 存储位置：知识地图配置表（knowledge_map_config）

**交互流程**
1. 管理员进入"知识地图配置"菜单
2. 查看已有地图列表
3. 点击"新增地图"打开配置表单
4. 填写地图名称和类型
5. 配置层级结构（选择分类维度）
6. 配置卡片展示字段（拖拽排序）
7. 设置适用角色
8. 保存配置
9. 前端实时渲染知识地图

**验收标准**
- [ ] 支持创建多个知识地图（按行业、按产品等）
- [ ] 层级结构支持拖拽调整顺序

#### 功能模块8：AI问答功能

**功能描述**
一线人员通过AI问答快速获取知识库内容，基于RAG（检索增强生成）模式，先从向量数据库检索相关文档，再由大模型生成结构化答案，支持5轮上下文保持。

**用户故事**
```
作为一线售前，
我希望能够用自然语言提问并获得准确的答案，
以便快速获取产品知识而无需翻阅大量文档。

作为一线交付，
我希望AI答案能够附带来源链接，
以便验证答案的准确性并查看详细内容。
```

**输入定义**
- 数据来源：用户的自然语言问题
- 格式要求：
  - 问题文本：最多500字
  - 上下文：支持多轮对话，保留前5轮对话的上下文
  - 检索参数：可调整检索的文档数量（默认3篇）

**输出定义**
- 结果形式：AI对话界面，包含答案和来源引用
- 答案结构：
  - 直接回答：针对问题的核心答案
  - 详细说明：展开说明的补充信息
  - 引用来源：关联的场景/解决方案/案例链接
  - 相关推荐：其他相关内容推荐

**技术流程**：
1. 用户输入问题
2. 系统将问题向量化
3. 在向量数据库中检索最相似的Top-K文档
4. 将问题和检索到的文档拼接成Prompt
5. 调用大模型API生成答案
6. 解析答案并格式化输出
7. 记录问答日志（用于优化检索效果）

**交互流程**
1. 用户在首页AI问答框输入问题
2. 系统显示"思考中..."动画
3. 返回AI答案（流式输出）
4. 答案下方显示引用来源（场景/解决方案/案例卡片）
5. 用户可点击来源查看详情
6. 支持多轮对话，上下文保持（最多5轮）
7. 用户可对答案点赞/点踩（用于反馈优化）

**验收标准**
- [ ] AI问答响应时间<3秒
- [ ] 答案必须附带来源引用
- [ ] 支持多轮对话，上下文保持（最多5轮）
- [ ] 支持答案反馈（点赞/点踩）
- [ ] 问答日志记录完整（问题、答案、引用、用户）
- [ ] 支持问答历史查看

#### 存储方案

- **关系型数据库**：PostgreSQL 15+
  - 存储所有结构化数据（场景、解决方案、案例、基础数据）
  - 支持JSON字段，存储灵活的元数据
  - 支持全文检索（PostgreSQL内置全文检索能力）
  - 支持UUID数据类型
  - 数据量预估：10万级（场景1万+解决方案5万+案例4万）

- **向量数据库**：Milvus 2.3+
  - 存储文档向量化数据
  - 支持相似度搜索（HNSW索引）
  - 支持本地部署，符合企业安全要求
  - 数据量预估：5万篇文档向量

- **对象存储**：MinIO / 企业OSS
  - 存储上传的文档（PPT、Word、PDF）
  - 支持大文件上传和断点续传
  - 存储空间预估：2TB（假设每个解决方案文档平均10MB）

- **缓存层**：Redis 7+
  - 缓存热点数据（知识地图、热门搜索词）
  - 缓存用户会话和API Token
  - 缓存AI问答上下文（5轮对话）

- **全文检索**：PostgreSQL内置全文检索
  - 如果性能不足，可扩展引入Elasticsearch
  - 当前阶段使用PostgreSQL全文检索功能

#### 数据流设计

```
[文档上传] → [文档解析] → [元数据提取] → [结构化存储]
                                      ↓
                               [文本向量化] → [向量数据库]
                                      ↓
                              [搜索索引更新]

### 3.2 系统架构

#### 架构模式
采用**前后端分离的分层架构**模式：

- 业务逻辑层：负责核心业务处理
- 数据访问层：负责数据存储和访问
- 向量检索层：负责向量相似度搜索
- AI服务层：负责大模型调用和RAG处理

```

**数据格式**
- 使用JSON格式
- 统一响应结构

**响应格式示例**：
```json
{
  "code": 200,
  "message": "success",
  "data": {
    "items": [...],
    "total": 100,
    "page": 1,
    "page_size": 10
  },
  "timestamp": 1701234567
}
```

**错误码规范**
| 错误码 | 含义 | 说明 |
|-------|------|------|
| 200 | 成功 | 请求成功 |
| 400 | 请求错误 | 参数错误或格式错误 |
| 401 | 未授权 | Token无效或过期 |
| 403 | 禁止访问 | 权限不足 |
| 404 | 未找到 | 资源不存在 |
| 429 | 超出限制 | 超出速率限制 |
| 500 | 服务器错误 | 服务器内部错误 |

### 3.3 非功能性需求

#### 性能要求
- **页面加载时间**：首屏加载 < 2秒
- **接口响应时间**：
  - 普通API响应 < 300ms（P95）
  - 搜索接口响应 < 500ms（P95）
  - AI问答响应 < 3秒
- **并发能力**：支持 500+ 并发用户
- **数据量**：支持 10万级数据存储
- **文档处理**：单个文档向量化时间 < 30秒

#### 安全要求
- **认证**：JWT Token认证，有效期2小时，支持刷新Token
- **授权**：基于角色的权限控制（RBAC）
- **数据加密**：
  - 传输使用HTTPS
  - API密钥加密存储
  - 敏感文档加密存储（可选）
- **防攻击**：
  - 防止XSS攻击（前端输入过滤）
  - 防止CSRF攻击（Token验证）
  - 防止SQL注入（ORM参数化查询）
  - 速率限制（防止暴力破解）

#### 兼容性要求
- **浏览器**：支持 Chrome、Firefox、Safari、Edge 最新版本
- **设备**：支持桌面端，移动端适配
- **分辨率**：适配 1920x1080、1366x768、移动端 320px-768px

#### 可维护性要求
- **代码规范**：遵循 ESLint/Prettier 规范
- **注释**：关键逻辑必须有注释
- **测试覆盖**：核心功能单元测试覆盖率 > 80%
- **日志**：关键操作记录日志（INFO、WARN、ERROR）
- **监控**：接口响应时间、错误率监控

#### 可扩展性要求
- **水平扩展**：支持多实例部署，通过Nginx负载均衡
- **数据库扩展**：PostgreSQL支持读写分离
- **向量库扩展**：Milvus支持分布式部署
- **缓存扩展**：Redis支持集群模式

#### 可用性要求
- **系统可用性**：99.5%（月度）
- **数据备份**：每日自动备份，保留30天
- **灾难恢复**：RPO < 1小时，RTO < 4小时
